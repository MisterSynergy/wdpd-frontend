About patrol

“Patrol” is a MediaWiki function that serves as a tool to coordinate countervandalism activities.

Edits in Wikidata by non-registered users (“IPs”) or newcomers with fewer than four days of participation and 50 edits are being marked with a “patrol” flag in the database for a period of 30 days. This should encourage experienced users to review these edits, as these edits are being highlighted in the Wikidata interface and some countervandalism tools. Most of the edits requiring “patrol” are fine; however, some new editors have difficulties to make acceptable edits due to a lack of experience, and there are also occasional “vandals” who make edits with bad faith that are potentially damaging to Wikidata’s reputation. The patrol process predominantly aims to remove vandalism as quickly as possible from the Wikidata project, and to remove good edits from the unpatrolled changes review workflow.

A difficulty in Wikidata’s patrol process is that the edits to patrol are usually relatively “atomic” in nature, i.e. they come without much context that would help to identify them as good-faith or bad-faith. The multilingual nature of the Wikidata project also exposes editors to edits in all sorts of languages. It is therefore advisable to filter recent changes by criteria that make it easier to patrol efficiently. Useful patrol strategies are outlined below.

In Wikidata, all users with “autoconfirmed” right can actively patrol unpatrolled contributions, and their own contributions are automatically patrolled. Further information about patrol can be found at mediawiki.org and in Wikidata itself (draft page).

Make use of additional patrolling tools

The basic Mediawiki software that serves Wikidata provides only a few rather inefficient patrolling opportunities in the default web user interface. In order to patrol efficiently, you should set up your account to use a couple of additional tools that nicely integrate into the user interface. We start with a selection of useful Javascript additions to your account:

  • The “Mark as patrolled” gadget from your Wikidata gadgets preferences
  • The “Request Deletion” gadget from your Wikidata gadgets preferences
  • patrolRevisions.js, to be installed in your common.js page at Wikidata
  • DiffLists.js, to be installed in your common.js page at Wikidata

Besides this, you can use the following tools on Wikimedia Toolforge:

  • reCh
  • Wikidata vandalism dashboard
  • Speedpatrolling
  • There are a couple of other recent changes filter tools, but most of them do not seem to be adapted to Wikidata’s filter needs.

Patrolling strategies

For efficient patrolling, it is useful to concentrate on one of these (or other) tasks:

Real-time recent changes patrolling

You can patrol changes nearly in real-time using Special:RecentChanges on Wikidata with tag and filter options. An alternative is to use one of the plenty real-time recent changes tools, however some of them are not particularly adapted for Wikidata. Here is a (potentially incomplete) list: ORC, Huggle, Twinkle, raun, RTRC, SWViewer, STiki, LiveRC, or Vandal Fighter.

The aim with this strategy is to skim the blatant vandalism and to remove it immediately.

Language or project based patrolling

Quite a lot of changes require language skills to patrol them. You can use tools such as reCh, WDVD or Speedpatrolling to patrol changes in a given language; as a side-task, you can also patrol unpatrolled sitelink changes on Wikimedia projects related to your language (use reCh for this task).

The unpatrolled changes volume of all languages except English can be patrolled by a single user, although for the larger ones (Spanish, German, French, Italian, Russian), three to five regular patrollers would be better to keep the backlog short. For English, however, significantly more patrollers are required due to the volume of changes to be patrolled.

This task does not need to be performed real-time, but clearing the backlog once a day would be a desirable goal.

User- and item-based patrolling

Finally, a significant amount of unpatrolled changes is made by relatively few users, or in larger numbers on a single item. Unlike the single revision-based patrol strategies above, this approach aims to identify whether a given unregistered or new user edits with good faith (and the required skills), in order to batch patrol all of their changes. This tool provides some worklists for exactly this strategy. The assessment and patrolling itself, however, are being done onwiki using the patrolRevisions.js user script. For an efficient assessment of a user’s contributions on their Special:Contributions page, the DiffLists.js user script is extremely helpful as it displays added and removed content at the same time.

This task also does not need to be performed real-time.
